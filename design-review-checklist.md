# Design Review Checklist

项目：《全国社区治理协同地图》

用途：所有页面、模块、组件、地图、AI 系统、数据层上线前的审美协议检查。

结论规则：

**只要出现任意禁止项，禁止进入系统。**

## 1. 页面级准入检查

上线前必须逐项确认：

- 是否像“城市 AI 操作系统”，而不是网站？
- 是否像“国家级治理中枢”，而不是企业后台？
- 是否拥有空气感？
- 是否拥有深空感？
- 是否拥有神经网络感？
- 是否拥有运行态？
- 是否拥有 AI 协同感？
- 用户第一眼是否会觉得：“这不是普通系统”？

如果任意一项回答为“否”，禁止上线。

## 2. 禁止项检查

页面不得出现：

- 普通后台感。
- 传统 dashboard 感。
- BI 大屏感。
- 廉价科技蓝。
- 赛博朋克。
- 高频闪烁。
- 信息密度过高。
- 失去空气感。
- 失去深空感。
- 失去国家级气质。

出现任意一项，禁止进入系统。

## 3. 视觉空间检查

页面必须具备统一空间层级：

- Layer 1：深空背景层。
- Layer 2：城市纹理层。
- Layer 3：神经网络层。
- Layer 4：AI 扫描层。
- Layer 5：实时治理数据层。
- Layer 6：前景交互层。

允许部分页面简化层级，但不能失去：

- 深色沉浸背景。
- 治理数据感。
- AI 感知或协同线索。
- 前景信息秩序。

## 4. 动效检查

所有动效必须符合：

- 慢。
- 克制。
- 呼吸感。
- 流体感。
- 深空感。
- 有明确系统状态含义。

禁止：

- 炫技。
- 发光过量。
- 动效过快。
- 高频闪烁。
- 快速旋转。
- 页面翻转。
- 粒子爆炸。
- 游戏 UI 动效。

## 5. 组件准入检查

新增组件前必须判断：

- 是否克制？
- 是否安静？
- 是否高级？
- 是否未来治理化？
- 是否属于同一视觉文明？
- 是否减少后台感？
- 是否保持信息真实可解释？
- 是否不依赖过多装饰？

如果组件只是“好看”但没有治理含义，不进入系统。

## 6. 地图与数据检查

地图必须表达：

- 城市节点。
- 社区热区。
- 风险扩散。
- 数据迁移。
- 协同连接。
- AI 扫描。

数据必须说明：

- 数据对象。
- 所属城市或区域。
- 当前状态。
- 来源或含义。
- 是否进入 AI 研判或协同流程。

禁止：

- 无意义节点。
- 无解释大数字。
- 彩色图表堆砌。
- 传统 BI 图表墙。

## 7. AI 模块检查

AI 模块必须像研判系统，而不是聊天玩具。

必须具备至少三项：

- 输入来源。
- 摘要状态。
- 关键证据。
- 风险判断。
- 关联案例。
- 建议动作。
- 协同流转状态。

禁止：

- 拟人头像。
- 客服聊天主界面。
- 魔法球。
- “AI 全自动解决”式文案。

## 8. 文案检查

必须使用“AI 文明语言”。

推荐：

- 城市运行感知。
- AI 协同网络。
- 治理神经节点。
- 风险扩散路径。
- 政策解析队列。
- 案例同步网络。
- 共建信号流。
- 全国治理信号。

禁止：

- 数据管理。
- 系统监控。
- 后台管理。
- 功能模块。
- 信息列表。

## 9. 最终审查结论

上线前必须给出结论：

```text
页面名称：
审查人：
是否通过视觉宪法：
是否存在禁止项：
是否符合统一世界观：
是否允许进入系统：
主要风险：
修正建议：
```

只有“是否允许进入系统”为“是”，才可继续实现或上线。
