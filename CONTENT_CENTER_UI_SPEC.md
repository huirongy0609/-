# CONTENT_CENTER_UI_SPEC

本文件定义 `内容中心` UI 规范。

本阶段仅输出设计规范，不涉及开发。

## 1. 内容中心定位

内容中心是平台的行业内容沉淀空间。

它承载：

```text
政策动态
实践案例
深度观察
人物访谈
参访纪实
活动报道
```

案例属于内容中心，不作为独立一级案例中心。

## 2. 视觉参考

参考：

- 财新：内容可信度、标题权重、阅读秩序。
- 得到：知识服务感、学习连续性。
- 36氪：行业内容组织、专题感。
- Notion：轻量结构、栏目组织、知识库感。

不参考：

- 传统CMS。
- 政府新闻门户。
- 后台列表。
- 数据表格。

## 3. 设计目标

内容中心应让用户感觉：

- 内容经过筛选和编辑。
- 平台有长期行业沉淀。
- 案例和政策能被持续阅读与复用。
- 用户能从内容进入制度、成长和资源连接。

## 4. 页面结构

建议结构：

```text
顶部：内容中心标题与定位
栏目导航：政策动态 / 实践案例 / 深度观察 / 人物访谈 / 参访纪实 / 活动报道
编辑精选：1个主内容 + 2-3个辅助内容
最新内容流：按时间更新
专题聚合：按主题组织
推荐阅读：基于用户画像或当前栏目
```

## 5. 栏目导航

栏目导航应轻，不应像后台 tab。

建议：

- 横向文字导航。
- 当前栏目用品牌主色或底部细线。
- 不使用厚重按钮。
- 不使用大面积色块。

## 6. 内容卡片

内容卡片字段：

```text
栏目
标题
摘要
标签
来源/作者
发布时间
适合对象
```

卡片原则：

- 标题优先。
- 摘要克制。
- 图片不是必须。
- 标签不应过多。
- 不显示过多统计数据。

## 7. 实践案例展示

实践案例不应像表格。

建议字段：

```text
标题
城市
治理主题
核心问题
治理方法
阶段成果
可复用经验
```

UI形式：

- 编辑化案例卡。
- 重点案例可使用长卡片。
- 列表案例使用紧凑阅读型卡片。

案例应与：

- 制度中心
- 治理实验室
- 成长世界任务
- AI信托助手

形成入口关系。

## 8. 深度观察展示

深度观察应有文章感。

UI形式：

- 较宽标题。
- 清晰摘要。
- 作者/来源信息。
- 阅读时间。
- 相关文章推荐。

不使用资讯流刷屏感。

## 9. 人物访谈展示

人物访谈应体现真实人和经验。

UI形式：

- 人物名称和角色清晰。
- 摘要突出观点。
- 可标注“来自物业 / 社区 / 业委会 / 研究者 / 街道”。
- 视觉可比普通内容更温暖。

## 10. 参访纪实与活动报道

重点体现现场感和可信度。

UI形式：

- 时间、地点、参与对象。
- 关键收获。
- 相关制度/案例/课程连接。

## 11. 内容详情页原则

阅读页应优先保证沉浸感。

结构：

```text
标题
摘要
来源与时间
正文
关键观点
相关制度
相关案例
相关成长任务
资源连接入口
```

正文宽度要适合阅读。

禁止：

- 两侧广告栏。
- 密集推荐。
- 大量浮动按钮。
- 后台式详情字段。

## 12. 搜索与筛选

筛选应服务阅读，不应像后台检索。

可筛选：

```text
栏目
主题
角色
城市
适用对象
发布时间
```

推荐使用轻量筛选条。

禁止：

- 多层高级筛选面板。
- 复杂表格筛选。
- 数据库管理感。

## 13. 空状态

空状态应提供下一步。

示例：

```text
暂未找到相关内容
可以尝试调整关键词，或使用 AI信托助手帮你找制度、案例和模板。
```

## 14. 与其他模块关系

内容中心应自然连接：

- 制度中心：从案例进入制度模板。
- 成长世界：从内容进入学习任务。
- 产品中心：从内容进入图书、训练营、工具包。
- 用户识别系统：根据身份推荐内容。
- AI信托助手：帮助找内容、总结内容、生成材料。
